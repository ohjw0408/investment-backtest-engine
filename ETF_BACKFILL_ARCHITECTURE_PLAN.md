# ETF Backfill Architecture Plan

## Purpose

This document defines the target architecture for a reliable ETF backfill system covering Korean-listed ETFs and the full U.S.-listed ETF/ETP universe.

The current backfill engine is useful as an early prototype, but it is not reliable enough for long-horizon simulations across diverse ETFs. It relies on broad category labels, a small static U.S. ETF list, incomplete index data, and simple proxy mappings. This can produce materially wrong results, such as treating two economically similar products differently because one has long U.S. history and the other has only a short Korean listing history.

The goal is not to fabricate long histories for every ETF. The goal is to decide, for every ETF, whether historical backfill is justified, how it should be done, and how trustworthy the result is.

## Core Principle

Every backfilled price or dividend record must answer these questions:

1. What source or model produced it?
2. Which proxy, index, holdings, or factor model was used?
3. What assumptions were applied for currency, dividends, hedge cost, leverage, or duration?
4. What confidence grade does the system assign?
5. Can the generated data be selectively deleted and regenerated later?

If those questions cannot be answered, the system should not silently backfill.

## Current State Summary

Important current files:

- `modules/backfill_engine.py`
- `modules/price_loader.py`
- `modules/index_loader_develop.py`
- `modules/div_yield_loader.py`
- `modules/retirement/data_preparer.py`
- `modules/retirement/synthetic_price_generator.py`
- `modules/retirement/ticker_stats_cache.py`
- `data/meta/kr_etf_list.csv`
- `data/meta/us_etf_list.csv`
- `data/meta/index_master.db`
- `data/price_cache/price_daily.db`
- `data/meta/symbol_master.db`

Current flow:

1. `PriceLoader.get_price()` downloads actual ETF data from yfinance.
2. For Korean 6-digit ETFs and known U.S. ETFs, it calls `BackfillEngine.backfill()` once per loader session.
3. `BackfillEngine` loads `kr_etf_list.csv` and `us_etf_list.csv`.
4. Korean ETFs use `INDEX_MAP`.
5. U.S. ETFs use `US_CATEGORY_MAP`.
6. A proxy index series from `index_master.db.index_daily` is scaled to the ETF listing-date price.
7. Korean unhedged U.S. market ETFs multiply the proxy by USD/KRW before scaling.
8. Dividends are injected quarterly from `index_div_yield` or from the ETF's own observed dividend yield.
9. Backfilled rows are stored in `price_daily` with `volume = 0`.

Observed weaknesses:

- `US_CATEGORY_MAP` is too coarse. Many unrelated ETFs collapse into S&P 500 or Nasdaq proxies.
- `INDEX_MAP` covers only a subset of Korean ETF index labels.
- `symbol_master.db.underlying_symbol` exists but is currently unused and empty.
- `DJUSDIV100` is mapped but the current index daily data may be missing or too short.
- `KOSDAQ150` has ETF rows in `kr_etf_list.csv`, but the existing index code is `KQ150`; the current map does not connect them.
- Bond ETFs are mapped to yield series such as `DGS10` and `DGS30`, which are not bond price or total-return indices.
- `index_loader_develop.py` calls `_fetch_fred()` but the FRED code is not a valid method in the current file structure.
- Backfill failures are still marked as attempted in `PriceLoader._backfilled_codes`, preventing retry in the same session.
- `volume = 0` is used as a weak proxy for generated rows, but it does not distinguish index backfill, synthetic backfill, or other generated methods.
- Existing generated rows cannot be safely audited, deleted by method, or regenerated by version.

## Target Outcome

The final system should support:

- Full U.S.-listed ETF/ETP universe ingestion, not just the current small curated list.
- Full Korean ETF universe ingestion.
- ETF-specific proxy mapping instead of broad keyword/category mapping.
- Confidence-graded backfill.
- Explicit no-backfill policy for unsuitable products.
- Separate backfill methods for index replication, parent ETF replication, holdings replication, regression proxy, bond duration modeling, leverage/inverse modeling, and synthetic fallback.
- Provenance tracking for every generated row.
- Automated validation gates before generated data is accepted.
- Reproducible migration and regeneration of existing generated histories.

## Confidence Grades

Use these grades consistently in UI, reports, APIs, and DB tables.

| Grade | Meaning | Allowed Use |
|---|---|---|
| A | Exact or near-exact backfill using official index, total-return index, or high-quality holdings replication | Long-horizon simulation allowed |
| B | Strong proxy such as a parent ETF or well-validated close index with low tracking error | Simulation allowed with notice |
| C | Regression or factor proxy with acceptable but imperfect fit | Use with warning |
| D | Synthetic/statistical history with weak ETF-specific evidence | Exploratory only |
| F | Unsafe or unsupported | No backfill |

The system should prefer no backfill over a misleading D/F-grade history.

## Data Sources

### U.S. ETF Universe

Preferred source priority:

1. SEC investment company series/class data for registered fund identity and class/ticker linkage.
2. Exchange listings from NYSE/Nasdaq/Cboe where available.
3. Issuer websites for holdings, fund inception, index name, expense ratio, and distributions.
4. yfinance only as a convenience source for prices and basic metadata, not as the authoritative metadata source.
5. ETF.com or other commercial/free screeners only as supplemental classification if license and access allow.

Implementation notes:

- Do not assume the U.S. ETF universe is fixed at 3,000. Build the collector so it can ingest all currently listed U.S. ETF/ETP symbols.
- Store the source and timestamp for each field where practical.
- Distinguish ETFs, ETNs, commodity trusts, crypto trusts, closed-end funds, and leveraged ETPs.

### Korean ETF Universe

Preferred source priority:

1. KRX Open API for ETF listing and daily data where available.
2. KRX ETF PDF/portfolio composition data for holdings and underlying metadata.
3. Issuer websites for official fund pages, underlying index, distribution policy, hedge policy, and fees.
4. FinanceDataReader/yfinance as convenience price sources.
5. Existing `kr_etf_list.csv` as seed data, not final truth.

Implementation notes:

- Korean ETF names often include strategy hints, but names must not be the final mapping authority.
- Use name-based inference only to propose mappings with `review_status = needs_review`.
- For Korean-listed overseas ETFs, store both trading currency and economic exposure currency.

## Backfill Method Hierarchy

For each ETF, try methods in this order. Stop at the first method that passes validation and meets the requested confidence threshold.

### Method 1. Actual Data

Use actual ETF prices and actual corporate actions from listing date onward.

Rules:

- Never overwrite actual rows with generated rows.
- Use split-adjusted or non-adjusted prices consistently with the rest of the engine.
- Store actual corporate actions separately from generated actions.

### Method 2. Exact Total-Return Index Backfill

Use when the ETF tracks a known index and official or reliable total-return series is available.

Examples:

- S&P 500 total return for S&P 500 ETFs.
- Nasdaq-100 total return for Nasdaq ETFs.
- KOSPI200 total return for KOSPI200 ETFs if available.
- Dow Jones U.S. Dividend 100 total return or price/yield combination for SCHD-like products.

Rules:

- Prefer total-return index for accumulation simulations.
- If only price index is available, handle dividends separately through `dividend_policy`.
- Scale the index to the ETF's first actual price after all currency adjustments.

Confidence:

- A if official total-return index is available and mapped exactly.
- B if close official price index plus reliable dividend yield is used.

### Method 3. Parent ETF Backfill

Use when a Korean-listed ETF is a local wrapper around a U.S. ETF or same benchmark.

Example:

- `TIGER 미국배당다우존스` and SCHD-like products should map to the same Dow Jones U.S. Dividend 100 proxy.

Rules:

- For U.S. ETF parent: keep USD proxy in USD.
- For Korean unhedged product: apply `USD/KRW`.
- For Korean hedged product: do not multiply spot FX directly; use hedge-cost model or mark lower confidence until model exists.
- Scale to local ETF first actual price.

Confidence:

- A if underlying index is identical and currency/hedge treatment is well modeled.
- B if parent ETF is close but not legally identical.

### Method 4. Holdings Replication

Use when holdings data exists before the ETF listing date or for overlapping validation.

Rules:

- Use actual holdings weights at known rebalance dates.
- If only recent holdings are available, backfill only to the earliest reliable holdings date unless using a lower-grade model.
- Rebalance according to available holdings snapshots.
- Use local price histories for constituents.
- For Korean ETFs holding U.S. stocks, apply FX per constituent exposure.

Confidence:

- A/B if holdings coverage is frequent and complete.
- C if holdings are sparse or inferred.

### Method 5. Regression Proxy Backfill

Use when actual overlapping ETF history is long enough to estimate a robust proxy model.

Candidate factors:

- Broad equity indices: S&P 500, Nasdaq-100, Russell 2000, MSCI World, MSCI EM, KOSPI200, KOSDAQ150.
- Sector indices: technology, semiconductor, healthcare, financials, energy, REITs.
- Style factors: value, growth, quality, dividend, low volatility, momentum.
- FX factors: USD/KRW, USD/JPY, etc.
- Rates and spreads for bond ETFs.

Validation requirements:

- Minimum overlap period, normally at least 24 months.
- Correlation above threshold.
- Tracking error below threshold.
- Stable beta across subperiods.
- Sensible economic interpretation.

Confidence:

- C by default.
- B only if extremely stable and economically obvious.

### Method 6. Bond Duration Model

Use for bond ETFs when no total-return bond index is available.

Do not use raw yield levels as prices.

Approximate return:

```text
bond_return ~= carry_income
              - duration * yield_change
              + 0.5 * convexity * yield_change^2
              - credit_spread_change * spread_duration
              - fees
              - hedge_cost_if_any
```

ETF types:

- Short Treasury
- Intermediate Treasury
- Long Treasury
- TIPS
- Aggregate bonds
- Investment-grade corporate
- High yield
- Municipal
- Emerging market bonds
- International bonds

Required data:

- Treasury curve series.
- Credit spread series for IG/HY if used.
- Inflation breakeven or real yield for TIPS if possible.
- Duration and convexity assumptions by ETF or category.
- Expense ratio.
- FX and hedge cost for Korean-listed overseas bond ETFs.

Confidence:

- B if a matching total-return bond index exists.
- C if modeled from rates and spreads with good validation.
- D if duration assumptions are broad.
- F if the ETF is complex and cannot be validated.

### Method 7. Leverage and Inverse Model

Use for daily-reset leveraged and inverse ETFs.

Rules:

- Apply daily leverage to underlying daily returns.
- Include financing cost, fees, and volatility drag.
- Reset daily, not monthly or annually.
- Validate against actual overlap.

Confidence:

- B/C if underlying is exact and validation passes.
- F if underlying is unknown or path dependency cannot be modeled.

### Method 8. Covered Call / Option Income Model

Use only when there is a tested model.

Rules:

- Covered call ETFs are not equivalent to their underlying index.
- Model must include option premium, capped upside, downside exposure, distribution policy, and fees.
- If no model exists, mark as F or D depending on the use case.

Confidence:

- C at best without detailed option data.
- F by default for automatic long-history backfill.

### Method 9. Synthetic Statistical Fallback

Use only when the user explicitly accepts low-confidence history or when the simulation only needs rough stress testing.

Rules:

- Store confidence D.
- Never present synthetic history as factual.
- Use `TickerStatsCache` only after excluding generated rows from the stats basis.
- Require minimum actual history.

## ETF Type Policies

### Plain Broad Equity ETFs

Examples:

- SPY, VOO, IVV
- QQQ, QQQM
- KODEX/TIGER/ACE/SOL U.S. S&P500
- KOSPI200 ETFs

Policy:

- Use exact or near-exact index.
- Apply currency policy for Korean overseas ETFs.
- Confidence A/B.

### Dividend Equity ETFs

Examples:

- SCHD
- TIGER/SOL/ACE/KODEX 미국배당다우존스

Policy:

- Map to Dow Jones U.S. Dividend 100 when verified.
- Use same proxy family for U.S. ETF and Korean wrappers.
- U.S. ETF stays USD; Korean unhedged products use USD/KRW.
- Dividend injection should use actual/proxy dividend yield history and observed distribution frequency.

Important:

- If the Dow Jones U.S. Dividend 100 index starts in 2002 and reliable daily data exists from 2002, Grade A/B backfill can start there.
- Earlier history should not be A-grade. Use a dividend-factor proxy only with lower confidence.

### Thematic / Active Equity ETFs

Examples:

- Tesla value-chain active ETFs
- AI active ETFs
- Robotics, battery, defense, nuclear, culture, metaverse themes

Policy:

- Do not map purely by name.
- First inspect holdings.
- If one stock dominates, a single-stock proxy can be considered.
- If holdings form a stable basket, use holdings replication.
- If holdings are unavailable, regression against candidate factors can propose a C-grade model.
- Active ETFs with low explanatory power should be no-backfill.

Example:

For a Tesla value-chain active ETF:

- Use TSLA only if observed holdings or returns prove TSLA dominance.
- Otherwise test TSLA, QQQ, auto/EV ETFs, USD/KRW, and sector factors.
- If validation fails, mark F.

### Bond ETFs

Policy:

- Never use yield levels as price series.
- Prefer total-return bond indices.
- Otherwise use duration/spread model.
- Store duration assumptions and model version.

### Money Market / Cash / CD / KOFR ETFs

Policy:

- Use short-rate accrual model.
- Price path should be low volatility and income-like.
- Do not use equity or bond price proxy.
- Korean CD/KOFR products need Korean short-rate data.

### Commodity ETFs

Policy:

- Physical gold/silver can use spot or futures proxy with clear caveat.
- Futures-based commodities need roll yield treatment.
- Crude oil futures ETFs should not simply use spot oil.

### Crypto ETFs

Policy:

- Use spot crypto price only for spot crypto ETFs/trusts.
- Futures crypto ETFs need futures/roll model.
- Confidence depends on structure.

### Leveraged / Inverse ETFs

Policy:

- Use daily reset model.
- Do not scale a long-term index directly by leverage over the full period.

**Broad-index leveraged/inverse (e.g., KODEX 200 선물 2X, TIGER 미국나스닥100 2X):**

- Underlying = known index (KOSPI200, NDX). Grade B/C after validation.
- Apply daily compounding: `r_lev[t] = leverage * r_index[t] - daily_fee`.
- Include volatility drag; long-term CAGR will diverge from `leverage * index_CAGR`.
- Validate correlation and max drawdown difference against actual overlap.

**Single-stock leveraged (e.g., 삼성전자 2X, TSLA 2X):**

- Underlying = single stock price from yfinance or KRX.
- Same daily reset formula; same volatility drag.
- Grade C max due to concentration and path dependency.
- If underlying stock data is unavailable pre-listing, Grade D or no-backfill.
- Korean regulatory expansion (2025~) will produce many of these; each must register underlying in `etf_proxy_map` individually.

**Inverse ETFs:**

- `r_inv[t] = -1 * r_index[t] - daily_fee`; same daily reset rule.
- Grade B/C if underlying is exact.

**All leveraged/inverse:**

- Never use monthly or annual scaling as substitute for daily reset.
- Store `leverage` coefficient and `underlying_symbol` in `etf_proxy_map`.
- Confidence F if underlying cannot be identified.

### Covered Call / Income Option ETFs

Policy:

- No automatic high-confidence backfill.
- Require option-income model or parent fund history.

### Regulatory Expansion ETFs (2025~ Korean Market)

Korean financial regulators relaxed ETF rules in 2025, enabling a wave of new product types. The backfill system must handle each category on arrival without manual per-ETF coding.

**Trigger:** When a new ETF appears in `symbol_master.db` with no `etf_proxy_map` entry, the system must:

1. Classify product type from ETF name + category metadata.
2. Look up `etf_proxy_map` for a matching proxy rule.
3. If no rule: assign `status = needs_review`, no backfill, no simulation blocker.
4. Alert operations queue for manual proxy assignment.

**Product types expected from Korean regulatory expansion:**

| Type | Example | Approach | Grade |
|---|---|---|---|
| Single-stock leveraged | 삼성전자 2X, SK하이닉스 2X | Method 7, underlying = KRX stock | C |
| Single-stock inverse | 삼성전자 -1X | Method 7, leverage = -1 | C |
| U.S. single-stock leveraged | TSLA 2X, NVDA 2X | Method 7, underlying = yfinance | C |
| Thematic active | 테슬라 밸류체인, AI반도체 | Method 9 regression or holdings | C/D/F |
| Covered call (KR) | KOSPI200 커버드콜 | Method 8, no auto high-confidence | D/F |
| Buffer/defined-outcome | 원금보장형 | No method yet; F until model exists | F |

**Key principle:** A surge of new ETFs must not require a developer to touch `backfill_engine.py` for each one. The `etf_proxy_map` table is the extension point. Adding a new ETF = inserting a row in `etf_proxy_map`, not changing code.

## Validation Gates

Generated backfill should be rejected unless validation passes where actual overlap exists.

Suggested metrics:

```text
correlation
tracking_error_annualized
mean_return_difference
volatility_difference
max_drawdown_difference
beta_stability
dividend_yield_difference
distribution_frequency_match
currency_return_attribution
```

Suggested starting thresholds:

| Metric | A/B Target | C Target |
---|---:|---:|
| Monthly correlation | >= 0.95 | >= 0.85 |
| Annualized tracking error | <= 3% | <= 8% |
| Volatility difference | <= 20% relative | <= 40% relative |
| CAGR difference over overlap | <= 2% annualized | <= 5% annualized |
| Dividend yield difference | <= 25% relative | <= 50% relative |

These are starting thresholds. They should be tuned by ETF type.

## Migration Plan

### Phase 0. Freeze and Diagnose

Goal:

Understand exactly what is currently generated before changing behavior.

Tasks:

- Add a diagnostic script, for example `tools/backfill_diagnostics.py`.
- Report each ETF's current status:
  - `actual_start`
  - `generated_start`
  - `first_real_date`
  - `volume_zero_rows`
  - current mapped proxy
  - index data range
  - dividend yield data range
  - whether current generated data is auditable
- Export reports:
  - `reports/backfill_status_kr.csv`
  - `reports/backfill_status_us.csv`
  - `reports/index_data_status.csv`
  - `reports/generated_rows_status.csv`

Acceptance criteria:

- Running diagnostics does not modify DB.
- Report identifies unsafe generated histories.
- Report clearly flags `DJUSDIV100`, `KOSDAQ150`, bond ETFs, and category-only U.S. mappings.

### Phase 1. Low-Risk Stabilization

Goal:

Stop obvious incorrect behavior without changing the full architecture yet.

Tasks:

- Fix `index_loader_develop.py` so `_fetch_fred()` exists as a real method.
- Add `KOSDAQ150 -> KQ150` to the current mapping.
- Repair or populate `DJUSDIV100` index data if a reliable source exists.
- Change `PriceLoader` so failed backfills are not marked as completed for the session.
- Add basic index data sufficiency checks:
  - reject index series with too few rows
  - reject index series whose start date is not before ETF listing date
- Add clear backfill status logging.

Acceptance criteria:

- SCHD-like and TIGER U.S. Dividend Dow Jones products no longer diverge because one uses only short actual history while the other uses a long proxy.
- KOSDAQ150 ETFs can map to KQ150 where appropriate.
- FRED-backed index loading does not fail due to missing method.

### Phase 2. Provenance Schema

Goal:

Make generated data auditable and reversible.

Tasks:

- Add DB migration script for:
  - `backfill_runs`
  - `price_daily_source`
  - `corporate_action_source`
  - optional `etf_proxy_map`
- Update backfill writes to also write provenance rows.
- Add deletion utility by `run_id`, `model_version`, or `confidence`.
- Add read helper to distinguish actual vs generated rows.

Acceptance criteria:

- Every newly generated price row has a provenance row.
- Generated rows can be deleted without touching actual data.
- Future stats calculations can exclude generated rows.

### Phase 3. ETF Universe Expansion

Goal:

Replace the tiny curated U.S. ETF list with a full universe pipeline.

Tasks:

- Build U.S. ETF universe collector:
  - SEC series/class data
  - exchange listings
  - yfinance fallback for prices only
  - issuer metadata where feasible
- Build Korean ETF universe collector:
  - KRX Open API
  - KRX ETF daily data
  - issuer page fallback
- Normalize into `etf_master`.
- Keep old CSVs as seed/export artifacts, not the final source of truth.

Acceptance criteria:

- U.S. universe count is not hard-coded and can exceed 3,000.
- Korean universe matches or exceeds current `kr_etf_list.csv`.
- Each metadata field has source and last-checked timestamp where practical.

### Phase 4. Proxy Mapping System

Goal:

Replace category-level mappings with ETF-level mappings.

Tasks:

- Create `etf_proxy_map`.
- Import current `INDEX_MAP` and `US_CATEGORY_MAP` as low-confidence seed mappings.
- Create high-confidence manual mappings for core ETFs:
  - S&P 500 family
  - Nasdaq-100 family
  - KOSPI200 family
  - KOSDAQ150 family
  - Dow Jones U.S. Dividend 100 family
  - major gold ETFs
  - major Treasury ETF families
- Add `review_status`.
- Build review report listing `needs_review` mappings.

Acceptance criteria:

- `BackfillEngine` reads `etf_proxy_map` first.
- Hard-coded maps are fallback only.
- A mapping can be rejected without changing code.

### Phase 5. Backfill Engine V2

Goal:

Refactor `BackfillEngine` into a dispatcher with independent methods.

Suggested structure:

```text
modules/backfill/
  __init__.py
  engine.py
  diagnostics.py
  models.py
  repository.py
  methods/
    exact_index.py
    parent_etf.py
    holdings.py
    regression.py
    bond_duration.py
    leverage_inverse.py
    synthetic.py
    no_backfill.py
```

Tasks:

- Add a `BackfillContext` object containing ETF metadata, proxy mapping, actual data range, index data range, and requested start/end.
- Add a common `BackfillResult`.
- Implement method selection by `proxy_level`.
- Add validation before saving.
- Save provenance for all generated rows.

Acceptance criteria:

- Current behavior can be reproduced for simple index ETFs.
- Method code is testable independently.
- Unsafe product types are rejected explicitly.

### Phase 6. Dividend Backfill V2

Goal:

Stop injecting all dividends as generic quarterly synthetic dividends.

Tasks:

- Create dividend policy module.
- Use actual distribution frequency when available.
- For proxy ETF backfill, use proxy ETF dividend schedule if appropriate.
- For index yield backfill, distribute according to ETF type and observed frequency.
- For no-dividend assets, do not inject dividends.
- Track generated dividend provenance.

Acceptance criteria:

- Monthly dividend ETFs receive monthly-like distributions where supported.
- Quarterly ETFs do not get monthly distributions.
- Covered call and income ETFs are not treated like ordinary equity dividend ETFs unless a model exists.

### Phase 7. Bond and Rate Models

Goal:

Replace invalid yield-as-price bond backfills.

Tasks:

- Add rate curve loader.
- Add spread loaders where possible.
- Define category-level duration assumptions.
- Add per-ETF duration overrides where data is available.
- Implement duration return model.
- Validate TLT, IEF, SHY, AGG, LQD, HYG against actual histories.

Acceptance criteria:

- Bond ETF backfills use yield changes, not yield levels as prices.
- Validation metrics are stored.
- Poorly validated bond products are C/D/F, not A/B.

### Phase 8. Holdings Replication

Goal:

Use holdings when index proxy is unavailable or insufficient.

Tasks:

- Store holdings snapshots.
- Normalize constituent identifiers.
- Load constituent prices.
- Rebalance by snapshot dates.
- Apply FX where needed.
- Validate against actual ETF overlap.

Acceptance criteria:

- Holdings replication can run on a small manually verified ETF set.
- Tracking error report is generated.

## Codex Review Notes: Automation Risks and Practical Rollout

This section records an implementation review added by Codex on 2026-05-28.

The proxy automation design is directionally correct, but it should be treated as an operations and review system, not as a fully automatic oracle. The safe target is:

1. Automatically classify and propose candidate mappings.
2. Automatically accept only narrow, validated A/B-grade families.
3. Default ambiguous products to `needs_review`.
4. Refuse to generate long histories when the system cannot explain the proxy, assumptions, confidence, and deletion path.

### Main Concerns

1. `underlying_symbol` is currently not a reliable input.
   - `symbol_master.db` has the column, but current data may be empty or incomplete.
   - Any leverage or single-stock automation that depends on this field must first populate and validate it.

2. Name/category inference is useful for triage, but unsafe as final truth.
   - Names such as TSLA, AI, robotics, semiconductor, income, or covered call can identify a product family.
   - They do not prove that a stock, sector index, or broad market index is a valid historical proxy.
   - Name/category-based mappings should be written with `review_status = needs_review` unless they belong to a narrow reviewed rule.

3. Current broad U.S. category mappings are dangerous if promoted into the new system unchanged.
   - Many U.S. ETF categories collapse into S&P 500 or Nasdaq proxies today.
   - These mappings may be useful as low-confidence seed data for diagnostics, but should not produce A/B-grade generated histories.

4. Leveraged ETFs require explicit underlying and daily-reset modeling.
   - The existing leverage multiplication is a start, but the new model should include daily reset, fees/financing, validation against overlap, and volatility-drag diagnostics.
   - Single-stock leveraged products must not be inferred only from ticker or name.

5. Thematic and active ETFs need holdings or regression validation.
   - If holdings are unavailable and regression is weak, no-backfill is better than a misleading long history.
   - A Tesla value-chain ETF should not map to TSLA unless holdings or return behavior proves TSLA dominance.

6. Covered call ETFs should remain rejected by default.
   - They are not equivalent to their underlying index.
   - A usable model must explain capped upside, option premium, downside exposure, distribution policy, and fees.

7. Provenance should come before broad expansion.
   - Without `backfill_runs`, `price_daily_source`, and `corporate_action_source`, wrong generated histories are difficult to audit, delete, or regenerate.
   - This is more important than adding many ETF types quickly.

### Practical Step-by-Step Rollout

#### Slice 1. Make Current Generated Data Observable

Goal: know what exists before changing behavior.

Tasks:

- Generate diagnostics for current actual range, generated range, volume-zero rows, mapped proxy, index range, and dividend source.
- Flag unsafe families: category-only U.S. mappings, bond yield-as-price mappings, covered call mappings, leveraged products with missing underlying, and thematic mappings with missing proxy data.
- Do not change generated rows in this slice.

Acceptance:

- A reviewer can see which ETFs are safe, suspicious, or unsupported.
- The report can be rerun without modifying DBs.

#### Slice 2. Add Minimal Provenance

Goal: make future generated rows reversible.

Tasks:

- Add `backfill_runs`.
- Add `price_daily_source`.
- Add `corporate_action_source`.
- Write provenance for newly generated rows only.
- Add delete-by-run helper before any large regeneration.

Acceptance:

- New generated rows can be deleted without touching actual price rows.
- Simulators can distinguish actual vs generated rows.

#### Slice 3. Introduce `etf_proxy_map` as the Extension Point

Goal: stop editing `backfill_engine.py` for every new ETF family.

Tasks:

- Create `etf_proxy_map`.
- Seed only reviewed core families at first:
  - S&P 500
  - Nasdaq-100
  - KOSPI200
  - KOSDAQ150
  - Dow Jones U.S. Dividend 100
  - major physical gold products if proxy data is sufficient
- Import old `INDEX_MAP` and `US_CATEGORY_MAP` only as low-confidence fallback or diagnostic seed rows.
- Add `review_status`.

Acceptance:

- `BackfillEngine` reads `etf_proxy_map` first.
- Hard-coded maps are fallback only.
- A bad mapping can be rejected by DB row, not by code change.

#### Slice 4. Add Explicit Reject Policies

Goal: prevent known-bad automatic backfills.

Tasks:

- Mark covered call, buffer/defined-outcome, unknown crypto futures, unknown thematic, and missing-underlying leveraged products as `proxy_level = none` or `needs_review`.
- Add user-facing status strings explaining why a product was not backfilled.
- Add tests that reject covered call ETFs without an option model.

Acceptance:

- JEPI/JEPQ-like products do not silently receive S&P 500 histories.
- TSLA/AI/thematic names do not automatically map to TSLA, QQQ, or S&P 500.

#### Slice 5. Implement Daily-Reset Leverage for Reviewed Underlyings

Goal: support broad-index and single-stock leveraged products only after the underlying is known.

Tasks:

- Store `underlying_symbol`, `leverage`, fee/financing assumptions, and confidence in `etf_proxy_map`.
- Load underlying stock/index histories from the correct source.
- Apply daily-reset compounding.
- Validate against actual overlap when available.

Acceptance:

- KOSPI200/Nasdaq-100 leveraged products can be C/B-grade after validation.
- Single-stock leveraged products are C-grade max and rejected if underlying is missing.

#### Slice 6. Add Holdings and Regression Only for Selected Families

Goal: support thematic/active ETFs without pretending weak proxies are factual.

Tasks:

- Start with a small reviewed ETF set.
- Store holdings snapshots where available.
- Build candidate factor returns.
- Fit and validate regression models.
- Store coefficients and validation summary.

Acceptance:

- A thematic ETF explains why a proxy was accepted or rejected.
- Weak models remain C/D/F and are not shown as high-confidence long histories.

#### Slice 7. Covered Call and Bond Models Later

Goal: defer complex models until the core provenance and mapping system is stable.

Tasks:

- Bond duration/spread model after total-return bond index options are exhausted.
- Covered call option-income model only after the simulator can represent capped upside, option premium, distribution policy, and fees.

Acceptance:

- Bond ETFs no longer use raw yield levels as price histories.
- Covered call ETFs are not backfilled automatically unless a tested model exists.
