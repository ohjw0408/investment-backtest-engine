## 0. 먼저 정할 것 (오너 결정)
1. **모델**: ① 월 구독(Pro) ② 일회성/평생 ③ 크레딧(시뮬 횟수). → 권장: **월 구독 단일 Pro** (단순·예측가능).
2. **무료 vs Pro 경계** (가치 가르기). 후보:
   - 무거운 MC/역산 **무제한**(무료는 일 N회 제한 — 보안플랜 rate limit과 직결).
   - **멀티계좌·세금 시나리오·고급 분석**(기여도·거시 오버레이) Pro 전용.
   - 저장 포트폴리오 개수, 알림 개수, 데이터 내보내기, 광고 제거(광고 도입 시).
   - ⚠️ 핵심 분석을 과하게 막으면 베타 신뢰 잃음 — "무료도 충분히 쓸만 + Pro는 파워유저" 선.
3. **가격**: 월 ₩(예 4,900~9,900). 연결제 할인 여부.

## 1. 결제 수단 (한국)
- **PortOne(구 아임포트)** 또는 **Toss Payments** 정기결제(빌링키). 카드 정기결제.
- PG 가맹 심사·사업자등록·전자결제 계약 필요(리드타임 있음 — 사업 절차).

## 2. 기술 설계
- `users` 확장: `plan TEXT DEFAULT 'free'`, `plan_started_at`, `plan_expires_at`, `billing_key`(PG 빌링키 토큰, 카드정보 직접 저장 금지).
- 신규 테이블 `payments`(결제이력: user_id·amount·status·pg_tx_id·created_at), `subscriptions`(상태·다음결제일).
- **구독 상태 게이팅**: 데코레이터 `@require_pro` — Pro 전용 엔드포인트/기능 보호. 프론트는 잠금 UI + 업그레이드 CTA.
- **rate limit 차등**: 보안플랜 2.1과 연계 — free는 빡센 한도, Pro는 완화(plan 기반 동적 한도).
- **정기결제 배치**: celery beat(이미 사용 중)로 만료 임박 빌링 실행 + 실패 재시도·해지 처리.
- **웹훅**: PG 결제 결과 수신 라우트(검증·멱등성).

## 3. 결제 관련 법적 (약관에 반영 — 법적플랜과 연계)
- 전자상거래법: 결제 전 가격·결제수단·해지방법 고지.
- 정기결제 고지(다음 결제일·금액·해지 안내).
- 청약철회·환불 정책(콘텐츠/디지털 서비스 기준).
- 세금계산서/현금영수증, 결제대행 표시.

## 4. 단계 (MVP → 확장)
- **M1 (MVP)**: 단일 Pro 월구독 1종. PortOne 빌링키 등록 → 정기결제 → `plan` 게이팅 1~2개 기능(예: 멀티계좌 + 무제한 MC). 결제이력·해지·환불 기본.
- **M2**: 연결제 할인, 무료 체험(N일), 업그레이드 퍼널·페이월 UX 다듬기.
- **M3**: 사용량 기반(크레딧) 또는 팀/가족 플랜 등 — 데이터 보고 결정.

## 5. 리스크
- PG 심사·사업자·계약 = 사업 리드타임(코드 외).
- 정기결제 규제(고지·해지 용이성) 위반 시 제재.
- 페이월 경계 잘못 = 베타 사용자 이탈. A/B·피드백 후 조정.
- 카드정보는 PG 빌링키만 보관(PCI 부담 회피) — 직접 저장 절대 금지.

## 6. 선결 의존
- 법적/보안 플랜 완료(약관·개인정보·환불·rate limit)가 결제의 **전제**. 그 위에서 M1 착수.
