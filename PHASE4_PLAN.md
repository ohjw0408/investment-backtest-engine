## 그룹 4A — 검색 & 종목 화면 개선 (우선순위 ★★★★★)

### A1. 검색 퍼지/오타 허용 강화

**현재 문제:**
- `SK하이닉스` → `sk 하이닉스` 검색 안 됨
- `skhynix` 검색 안 됨
- 한글/영문 혼용 전혀 처리 안 됨

**구현:**
- `modules/info_engine.py` → `search_fuzzy()` 개선
- 전처리: 공백 제거, 소문자화
- 토큰 분리 검색: `"sk 하이닉스"` → `["sk", "하이닉스"]` 각각 AND 검색
- rapidfuzz 편집거리 기반 fallback (수동 별칭 사전 불가 — 종목 수만 개 유지 못 함)
- `symbol_master.db`에 `name_en` 컬럼 추가 → yfinance `info['longName']`으로 자동 채움
  - 한국 종목: yfinance `.KS` 티커로 영문명 취득 (`삼성전자` → `Samsung Electronics Co., Ltd.`)
  - "애플" → "Apple" 근음 매칭은 rapidfuzz 편집거리가 커버
  - 완전 일관성은 yfinance 데이터 품질에 달림 (null 있을 수 있음)
- `symbol_master.db`에 `status`(active/delisted), `delisted_at` 컬럼 추가
  - 정기 배치 업데이트: yfinance 조회 실패 종목 → 자동 delisted 마킹
  - ETF 신규 상장/폐지도 같은 메커니즘으로 처리

### B4. 매수매도 트래킹 탭 + 추가매수 고도화

**목적:** 거래 내역 관리 + 추가매수 탭 매도 최소화 알고리즘 완성

**현재:** 추가매수 탭은 추가금액 → 그룹별 배분까지 구현됨. 매도 최소화 알고리즘 미구현.  
**미구현:** 거래 트래킹 탭 전체, `transactions` 테이블.

**B4-a: 매수매도 트래킹 탭 신규**
- `transactions` 테이블: (user_id, date, ticker, quantity, price, type[buy/sell], account_type)
- `myassets.html`에 "거래내역" 탭 추가
- 거래 입력 폼: 날짜, 종목, 수량, 가격, 계좌
- 거래 내역 목록 + 삭제

**B4-b: 추가매수 매도 최소화 완성**
- 현재: 단순 목표비중 비례 배분
- 추가: Cash-First Rebalancing 완전 구현



알고리즘 (Cash-First Rebalancing):
1. 추가 매수금 + 현재 포트 총액 = 리밸런싱 후 목표 총액
2. 목표 총액 × 목표 비중 = 각 종목 목표금액
3. 목표금액 > 현재금액인 종목: 매수 필요
4. 추가 매수금으로 매수 종목 커버 가능하면 → 매도 0
5. 추가 매수금으로 부족하면 → 비중 초과 종목만 최소 매도

UI:
- "추가 매수금" 입력란 (0원 기본값)

## 그룹 4F — Celery 2-worker 진행상태 UX 수정 (우선순위 ★★★★)


### F1. 대기 순위 → 실제 대기 여부 반영

**현재 문제:**
- 대기 순위 1번 → "1명 앞에 대기 중" 표시
- 그런데 워커가 2개니까 순위 1~2번은 즉시 처리 가능 → "곧 시작" 이어야 함
- 순위 0번만 "계산 중", 1번부터 전부 "대기 중"으로 표시하는 게 잘못됨

**수정 내용:**
- 대기 순위 `rank < concurrency(=2)` 이면 → "곧 시작됩니다" 또는 "계산 중 (워커 할당 완료)"
- `rank >= concurrency` 이면 → "대기 중 (내 앞 {rank - concurrency + 1}명)"
- 프론트엔드: `queue_pos`와 함께 `concurrency` 값을 API 응답에 포함하거나 하드코딩(2)

**구현:**
- `app.py` 상태 조회 엔드포인트 → `active_workers: 2` 포함
- `static/js/calculator.js` 대기 문구 로직 수정
- `tasks.py`의 `get_active_tasks()` 활용

**난이도:** 낮음 (0.5~1일)

---

### F2. 예상 대기시간 계산 수정

**현재 문제:**
- 예상 대기시간 = `평균 시뮬 시간 × 앞 순위 수`
- 2개 워커니까 → `평균 시뮬 시간 × ceil(앞 순위 수 / 2)` 로 보정 필요

**구현:**
- `tasks.py` `get_avg_duration()` 반환값 활용하는 계산식 수정
- 프론트엔드에서 `Math.ceil(queue_pos / concurrency) × avg_duration`

**난이도:** 매우 낮음 (0.5일)

> **결정:** 예상 시간 보정 안 함. 보수적으로 표시하는 게 사용자 경험상 낫다.

### E3. _price_cache TTL + LRU 관리

**현재 문제:**
- `_price_cache`는 딕셔너리로 서비스 재시작 전까지 무제한 누적
- 오늘 일찍 캐시된 데이터는 장 마감 후에도 갱신 안 됨 (당일 가격 지연)
- 종목 수천 개 누적 시 워커당 수백 MB 점유

**구현:**

**E3-a: TTL 만료**
- `_price_cache` 구조 변경: `{key: df}` → `{key: (df, cached_at, access_count)}`
- 접근 시 `time.time() - cached_at > TTL` 이면 재조회
- TTL 기본 60분, 암호화폐(CRYPTO) 5분
- 효과: 장 마감 후 1시간 이내 최신 가격 자동 반영

**E3-b: LRU 기반 용량 제한**
- 최대 엔트리 수: 300개 (종목당 ~500KB × 300 = ~150MB/워커)
- 300개 초과 시 `access_count` 낮은 것부터 20% 삭제 (evict 60개)
- `access_count`는 캐시 히트마다 +1

**구현 범위:** `modules/price_loader.py` `get_price()` 수정만으로 완결
**외부 라이브러리:** 불필요 (Python stdlib)

**난이도:** 낮음 (0.5~1일)

### E2. 코드 속도 최적화

**현재 병목 분석 필요 항목:**
- `AccumulationAnalyzer.run()` — while 루프로 수백 번 포트폴리오 시뮬
- `DividendSimulator._run_rolling()` — rolling window마다 전체 시뮬
- `RetirementPlanner` — 몬테카를로 시뮬레이션

**최적화 옵션 (난이도 순):**

| 방법 | 효과 | 난이도 | 리스크 |
|------|------|------|------|
| pandas 연산 벡터화 | 2~5x | 낮음 | 낮음 |
| numpy 직접 연산 | 3~10x | 중간 | 낮음 |
| numba `@jit` | 5~50x | 중간 | 중간 |
| C extension (Cython) | 10~100x | 매우 높음 | 높음 |



**권장 전략:**
1. profiling 먼저 (cProfile) → 실제 병목 확인
2. pandas → numpy 벡터화 (가장 안전)
3. numba `@jit` 핵심 루프에 적용
4. C extension은 ROI 낮으니 마지막

**난이도:** 중간~높음, 회귀 리스크 있어서 단위 테스트 먼저

## 별도 항목 — 한국 ETF 실보수 계산

**요청:** 한국 ETF의 실제 총보수율(총보수 + 기타비용) 표시

**현실:**

한국 ETF 실보수는 공식 API가 없음. 데이터 출처:
- KRX 정보데이터시스템 (비정형)
- 금융투자협회(KOFIA) 통합공시 — 스크래핑 가능하나 구조 자주 바뀜
- 각 운용사 홈페이지 (삼성, 미래에셋, KB, 한투 등)

**실현 가능한 방법:**
1. **수동 DB 구축:** 주요 100개 ETF 보수율 수동 입력 → `symbol_master.db`에 `expense_ratio` 컬럼 추가 → 분기별 수동 갱신
2. **KOFIA 스크래핑:** 자동화 가능하나 깨지기 쉬움
3. **ETF 평가사 API:** FnGuide, KIS, Quantiwise 등 — 유료

**권장:** 우선 수동 DB로 주요 ETF 100개만 처리. 나중에 자동화 검토.

**난이도:** 낮음(수동) ~ 높음(자동). 데이터 품질이 진짜 문제.
