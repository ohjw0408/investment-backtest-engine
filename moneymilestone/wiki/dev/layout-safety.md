# 레이아웃 안전성 — 규칙 · 검증 · 잔여

좁은 화면(폴드 바깥·플립 커버)과 **시스템 글자배율**에서 글자가 잘리거나 화면 밖으로
나가지 않게 하는 규칙과, 그걸 지키는지 검사하는 방법.

발단은 2026-07-22 오너 보고 — 어머니 폰(갤럭시 A)에서 홈 총자산이 잘려 보임.
해상도 문제가 아니었다. 자세한 경위는 [[../log]] 2026-07-22 항목.

---

## 1. 왜 생기는가 — 원인은 패턴 4개뿐

전수 검사에서 나온 실패 468건을 추적하니 개별 버그가 아니라 네 가지의 복제였다.

| # | 패턴 | 왜 터지나 |
|---|---|---|
| 1 | **`min-width: auto`** (flex/grid 자식 기본값) | "내용보다 좁아지지 않음" 이라 축소를 거부 → 컨테이너가 화면 밖으로 밀려남. **가장 큰 원인** |
| 2 | **`flex-wrap` 누락** | 고정폭 요소 + `nowrap` 텍스트가 한 줄에 갇혀 넘침 |
| 3 | **줄바꿈 불가 토큰** | 금액(`₩75,808,745`)·긴 URL. CSS로는 해결 불가 |
| 4 | **안 접히는 고정 여백** | 좁아져도 안 줄어드는 margin·padding |

추가로 **Android WebView 는 시스템 글자배율을 `textZoom` 으로 CSS 폰트에 곱한다.**
CSS 미디어쿼리로는 감지도 대응도 불가능한 축이다. 배율을 강제로 되돌리면
(`setTextZoom(100)`) 접근성 설정을 무시하는 것이라 **하지 않는다** — 대신 레이아웃이
큰 글자를 흡수하게 만든다.

> 참고: **해상도는 변수가 아니다.** CSS 는 밀도 독립 단위로 동작한다.
> 어머니 폰과 오너 폰이 똑같이 1080x2340 인데 한쪽만 깨졌던 게 그 증거.

## 2. 지켜야 할 규칙

### 전역 (이미 `static/css/style.css` 에 적용됨)

```css
*, *::before, *::after { min-width: 0; }     /* 패턴 1 — 468건 중 대부분을 이것 하나가 해소 */
body { overflow-wrap: break-word; }          /* 패턴 3 — 긴 URL */
img, canvas, svg, video, table { max-width: 100%; }
```

### 새 코드를 쓸 때

1. **grid 열은 `minmax(0, 1fr)`.** 그냥 `1fr` 은 `minmax(auto, 1fr)` 이라
   위 전역 규칙으로도 안 잡히는 **유일한 예외**다.
2. **고정 `width` 금지.** `max-width: min(340px, 100%)` 형태로.
3. **`min-width: Npx` 는 `min(Npx, 100%)`** 로. 전역 규칙보다 특이성이 높아
   그냥 쓰면 안전망을 무력화한다. (아이콘 크기 같은 작은 값은 무해)
4. **`nowrap` 이 있는 컨테이너에는 `flex-wrap: wrap`** 을 같이.
5. **중앙정렬 래퍼에는 `width: 100%` 를 반드시 함께.**
   `max-width: N; margin: 0 auto` 만 있으면 grid 아이템일 때 auto 마진이 stretch 를
   꺼서 **fit-content** 가 되고, 내부 min-content 를 그대로 물려받아 밀려난다.
   (simple·taxswitch·myportfolios·legal 4종이 전부 이 케이스였다)
6. **줄바꿈 불가 금액은 `white-space: nowrap` + `mmFitText`.**

### `mmFitText` (`static/js/base_app.js`)

넘칠 때**만** 하한(기본 16px)까지 폰트를 줄인다. 안 넘치면 손대지 않으므로 평상시
디자인이 그대로 유지된다. 배율과 싸우지 않는 게 원칙 — 잘려서 안 보이는 것보다
조금 작아도 다 보이는 게 낫다는 판단.

⚠️ **가용 폭은 즉시 부모가 아니라 "실제로 자르는 조상까지의 거리"로 잰다.**
부모가 이미 카드 밖으로 밀려나 있으면 부모 폭이 카드보다 넓게 나와 "맞는다"고
오판한다(내 자산 히어로에서 실측 343px vs 카드 236px).

## 3. 검사 방법

```bash
# 로컬 (로그인 화면까지 보려면 쿠키 필요)
venv\Scripts\python.exe tests\mint_session.py        # 쿠키 발급
venv\Scripts\python.exe tests\seed_e2e_holdings.py   # 보유종목 시드(빈 화면 방지)
node tests/test_layout_safety.js http://127.0.0.1:5000 <쿠키>

# prod
node tests/test_layout_safety.js https://moneymilestone.co.kr

# 옵션
#   --full    모서리 단계 건너뛰고 처음부터 전체 매트릭스
#   --shots   통과·실패 무관하게 스크린샷 저장(육안 검토용)
```

**범위**: 22페이지 × 뷰포트 8종(260~1280px) × 글자배율 4단(1.0~1.5) × 로그인·비로그인.

**2단계 전략**: 넘침은 거의 단조적(좁을수록·글자 클수록 심함)이라 가혹한 "모서리"만
먼저 돌리고, 걸린 페이지만 전체 매트릭스로 확대한다. 1,600 조합 → 200 남짓, 33분 → 8분.

### 검출기가 일부러 무시하는 것 (전부 실측으로 확인된 오탐)

건드리기 전에 반드시 읽을 것. 이 예외들을 빼면 오탐이 압도적으로 많아진다.

| 무시 대상 | 근거 |
|---|---|
| `scrollWidth` 기준 자체 | 장식용 `::after`(right:-50px)·`ellipsis`·네이티브 `<select>` 를 전부 버그로 셌다(**64건**). 그래서 **텍스트 노드 Range 사각형** 기준으로 판정 |
| 접힌 아코디언 (`max-height:0`) | 닫아둔 패널은 내용이 잘려 있는 게 정상. 이 오탐만 **1,100건 초과** |
| `text-overflow: ellipsis` · `-webkit-line-clamp` | 의도된 말줄임 |
| `position:absolute` + `pointer-events:none`/`user-select:none` | 워터마크 같은 비상호작용 장식 — 클릭도 선택도 안 되면 읽어야 할 내용이 아니다 |
| 가로 스크롤 컨테이너 안쪽 | 스크롤로 볼 수 있음 |

### 글자배율 시뮬레이션 함정 2개

`test_layout_safety.js` / `test_font_scale_responsive.js` 의 `applyTextZoom()`.

1. **반드시 2패스.** 1패스로 부모에 폰트를 박은 뒤 자식의 computed 를 읽으면
   상속 사슬을 따라 배율이 **복리**로 곱해진다(40px → 52 → 67.6). 실제 textZoom 은 복리가 아니다.
2. **인라인이 아니라 스타일시트로 주입.** 인라인으로 박으면 `mmFitText` 가 자연 크기를
   재려고 인라인을 지우는 순간 배율까지 사라져, 실제보다 작게 재고 **없는 버그를 보고**한다.

## 4. 이 검사가 보지 못하는 것

- **실기기가 아니다.** 실제 Android WebView 의 textZoom 과 완전히 같다는 보장 없음.
  실기기 확인은 어머니 폰(큰 배율 케이스) 1대뿐.
- **"안 깨짐"만 본다.** 안 잘렸지만 찌그러져 읽기 힘든 상태는 통과한다.
  그건 `--shots` 로 뽑아 사람이 봐야 한다.
- **화면 세로 비율·폴드 힌지·접힘 전환·RTL** 미검사.
