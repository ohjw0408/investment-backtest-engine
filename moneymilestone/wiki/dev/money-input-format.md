# 금액 입력란 콤마 표시 (`mmMoneyInput`)

평단가·초기투자금 같은 **금액 칸**에서 타이핑하는 동안 화면에 `1,980,000` 으로 보이게
한다. 저장·계산에 쓰이는 값은 그대로 `1980000`.

구현: `static/js/base_app.js` 하단 · 도입 2026-07-23 (오너 요청)

---

## 왜 "간단한 수정"이 아닌가 — 먼저 읽을 것

`<input type="number">` 는 **콤마를 담지 못한다.** 명세상 값이 유효한 부동소수점
문자열이어야 하고, 콤마가 들어가면 브라우저가 값을 빈 문자열로 취급한다.
CSS·포맷터로 우회 불가.

그래서 `type="text"` 로 바꿀 수밖에 없는데, **그 순간 그 칸을 읽는 모든 코드가
`"1,980,000"` 을 받는다.**

```js
parseFloat("1,980,000")   // → 1
```

**예외도 안 나고 계산도 정상적으로 돌아간다. 결과만 조용히 틀린다.**
이 앱은 세금·백테스트·은퇴 시뮬레이션이라 가장 위험한 종류의 버그다.

---

## 어떻게 해결했나 — 읽기 지점을 안 건드리는 방식

`.value` 를 읽는 지점이 앱 전체에 **400곳 가까이** 흩어져 있다. 하나씩 고치면
반드시 빠뜨리고, 빠뜨리면 위처럼 조용히 틀린다.

그래서 **해당 엘리먼트에만 `value` 접근자를 덮어썼다.**

```js
Object.defineProperty(el, 'value', {
  get() { return raw(desc.get.call(this)); },   // 읽기 = 항상 콤마 없는 숫자
  set(v) { desc.set.call(this, fmt(v)); },      // 쓰기 = 자동으로 콤마 표시
});
```

결과:

- 화면에는 콤마가 보이고, `el.value` 로 읽으면 언제나 숫자만 나온다
- **기존 읽기/쓰기 코드를 한 줄도 바꾸지 않았다**
- 앞으로 추가될 코드도 자동으로 안전하다
- 쓰기도 setter 가 받으므로 `el.value = h.avg_price`(기존 종목 수정)도 콤마가 붙는다

### 적용 전제 — 바꾸기 전에 확인한 것

`type=number` 에 묶인 기능이 하나라도 쓰였으면 이 방식이 깨진다. 앱 전체 **0건** 확인:

| 확인 대상 | 결과 |
|---|---|
| `valueAsNumber` | 0건 |
| 네이티브 `<form>` 전송 | 0건 |
| `stepUp()` / `stepDown()` | 0건 |
| `checkValidity()` / `:invalid` | 0건 |

**새로 이 중 하나를 쓰려면 금액 칸에서는 동작하지 않는다는 걸 알고 써야 한다.**

## 대상 27개 — 원 단위만

목록은 `window.MM_MONEY_INPUT_IDS` (`base_app.js`)가 단일 진실원천.
새 금액 칸을 만들면 **거기에 id 를 추가**하면 된다. 모달 등 나중에 그려지는 칸은
`window.mmApplyMoneyInputs()` 를 호출.

전체 숫자 입력란 73개 중 나머지는 **의도적으로 제외**:
수익률·물가상승률·수수료율 22개 · 기간(년) 11개 · 나이(세) 6개 · 수량(주) · 스텝 개수.
`7.5%` · `30년` · `만 58세` 에 콤마가 낄 일이 없다.

### ⚠️ ISA 전환 2개는 일부러 제외

`tsCurrentValue`(현재 평가액) · `tsCostBasis`(총 취득가).

`/tax-switch` 는 종목을 넣어도 계산이 자동으로 돌지 않는다 — POST 0건, 진행바
40초 대기에도 무응답. 그래서 골든 마스터로 **결과 보존을 검증할 수 없었다.**
**검증할 수 없으면 바꾸지 않는다**는 원칙으로 남겨뒀다.
그 페이지 계산을 자동으로 돌릴 수 있게 되면 함께 처리할 것.

## 안전장치 — `tests/golden_number_inputs.js`

**금액 칸을 건드릴 때는 반드시 이걸 먼저 돌린다.**

```bash
node tests/golden_number_inputs.js record http://127.0.0.1:5000 <쿠키>   # 변경 전
# ... 코드 변경 ...
node tests/golden_number_inputs.js verify http://127.0.0.1:5000 <쿠키>   # 변경 후
```

각 계산기에 종목을 심고 큰 금액(19,800,000)을 넣어 실행한 뒤 기록·대조한다.

| 기록 항목 | 왜 |
|---|---|
| **서버로 나가는 POST 본문** | 입력값이 그대로 반영되므로 "타이핑한 금액이 서버에 온전히 갔는가"를 직접 본다. 시세 같은 외부 변동에 오염되지 않는다. 콤마가 망가뜨리면 `initial_capital: 19800000` 이 `1` 이 되어 즉시 걸린다 |
| **화면에 렌더된 숫자** | 간편계산기처럼 서버를 안 타는 화면용 |

비결정 값(시세·시각)은 baseline 을 **2회** 돌려 두 번 다 같은 것만 골든으로 채택해
자동 제거한다.

### 하니스를 만들며 잡은 결함 3개 — 재사용 시 주의

1. **종목이 0개면 계산 버튼이 아무 일도 안 한다**(POST 0건).
   계산을 안 돌리는 골든 마스터는 아무것도 보증하지 못한다. 페이지별 종목 추가
   함수를 호출해 시드해야 한다.
2. **비중을 직접 조작하면 오히려 상태가 깨진다.** 60/40 으로 맞췄더니 백테스트가
   실행되지 않았다. 첫 종목이 자동 100% 라 조작 자체가 불필요.
3. **readback 을 `.value` 원문으로 기록하면 안 된다.** 콤마를 켜면 `"19,800,000"` 이
   되어 **설계상 무조건 실패**하는 무의미한 가드가 된다. 콤마를 제거한 숫자로 기록해야
   변경 전후가 비교 가능하다. (보고 싶은 건 "자릿수가 살아있는가"지 표기 형식이 아니다)
